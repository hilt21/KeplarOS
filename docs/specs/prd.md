# KEPLAR 产品需求文档

## 1. 产品概述

KEPLAR 是一个以目标空间和看板为协作总线的多智能体任务平台。它面向真实任务交付场景，将需求、拆解、执行、评审、回流和归档串成一条可追踪、可治理、可复用的协作链路。

系统的核心思路是**"统一目标空间 + 角色化工作视图"**。所有角色围绕同一目标上下文协同推进，但不同角色看到的界面、权限和操作入口不同，从而兼顾透明性与执行效率。

---

## 2. 背景与问题

当前团队在需求到交付的过程中，通常存在以下问题：

1. 目标描述分散在聊天、文档和会议记录中，难以形成统一上下文。
2. 需求、设计、实现和评审之间频繁切换，协调成本高。
3. 任务状态、责任人、阻塞原因和评审结果缺少统一可视化。
4. AI 能帮助执行，但如果缺少治理机制，容易出现结果不可控、异常难追踪的问题。
5. 任务完成后的产出难以沉淀为可复用资产。

**KEPLAR 试图解决的不是单一环节提效，而是把目标驱动的多角色协作变成一个持续运转的交付系统。**

---

## 3. 产品目标

### 3.1 产品愿景

为项目团队提供一个以看板为协作总线的多智能体任务平台，让需求、设计、实现和评审过程透明可控，减少上下文切换和人工协调成本。

### 3.2 核心目标

1. 将自然语言目标快速转化为可执行任务。
2. 让任务在不同节点和角色之间清晰流转。
3. 让 AI 负责高频、重复、结构化的加工工作。
4. 让人工只介入关键决策、异常处理和最终确认。
5. 让每个任务的执行过程可追踪、可审计、可复盘。

---

## 4. 目标用户

### 4.1 发起人（Initiator）

发起人通常是项目经理（对应 OE Project）或产品经理（对应 IP Project）。

**职责包括：**

- 定义目标、范围、约束和验收标准
- 创建目标空间和发起节点工作视图
- 跟踪关键任务状态和风险
- 对关键结果做人工确认

### 4.2 链路节点用户（Chain Users）

链路节点用户是各节点的实际执行和治理人员，包括：

- 节点主办者，用于管理节点任务进度
- 节点主管者或评审者，用于检查实现是否符合规范
- 运维或系统管理员，用于监控流程健康和异常状态

**职责包括：**

- 认领和处理节点卡片
- 补充上下文和本节点约束
- 推进任务执行
- 处理阻塞和异常回流
- 确认节点输出

### 4.3 AI 协同角色

系统内部的智能体角色包括：

- Backlog Refiner
- Todo Orchestrator
- Dev Crafter
- Review Guard
- Done Reporter
- Blocked Resolver

**职责包括：**

- 对卡片进行结构化加工
- 生成任务故事、执行简报、实现证据和评审意见
- 识别阻塞、缺失和风险
- 协助推进任务向下游流转

---

## 5. 产品范围

### 5.1 核心范围

1. 目标空间与节点看板管理
2. 自然语言目标转结构化故事
3. 卡片创建、流转与状态追踪
4. AI 泳道协同处理
5. 人工确认与异常回流
6. 外部系统集成
7. 执行轨迹追踪与审计

### 5.2 非核心范围

1. 不做通用项目管理工具替代品
2. 不做纯聊天机器人
3. 不做复杂 BPM 审批引擎
4. 不强制所有节点使用同一套页面操作逻辑

---

## 6. 核心概念定义

### 6.1 目标空间（Goal Space）

围绕一个业务目标形成的统一上下文容器。它承载目标描述、输入信息、约束、依赖、验收标准和整体进度。

### 6.2 节点看板（Node Board）

目标空间在具体节点下的工作视图。不同节点有各自的任务卡片、上下文、执行状态和责任人。

### 6.3 卡片（Card）

系统中的最小任务单元，用于承载需求、执行过程、结构化产出和审计信息。

### 6.4 泳道（Lane）

卡片在不同 AI 专家角色之间流转的处理阶段。每个泳道承担一种明确的加工职责。

### 6.5 结构化故事（YAML Story）

系统根据用户自然语言目标生成的结构化任务描述，包含问题陈述、验收准则、约束、依赖和输出要求。

---

## 7. 用户场景

### 7.1 发起人场景

发起人输入一句自然语言目标，例如"株机厂有一个10列6编组的南京地铁42号线项目开工了，交期是2027年6月，制动系统方案参考南京地铁24号线的技术方案。。。"，系统自动生成结构化故事，并创建初始卡片和看板。

### 7.2 节点主办者场景

链路节点用户进入本节点看板，认领上游卡片，查看输入是否完整，补充本节点约束，推动任务进入执行和评审。

### 7.3 评审者场景

评审者查看 Review 阶段输出，系统自动检查每条验收标准是否满足，缺失项自动退回并给出评审意见。

### 7.4 运维场景

运维人员监控会话成功率、接口失败率、阻塞任务数量和异常回流次数，及时处理系统健康问题。

---

## 8. 功能需求

### 8.1 目标空间与节点看板管理

系统应支持：

1. 创建目标空间
2. 创建和切换节点看板
3. 在不同角色视图下展示同一目标空间的不同工作视图
4. 展示卡片状态、依赖关系和责任人
5. 支持按节点、状态、优先级、风险筛选卡片

**关键要求：**

- 发起人看到的是目标和整体进度
- 链路用户看到的是本节点任务和上下游依赖
- AI 处理界面不应污染普通用户主视图

### 8.2 自然语言任务定义

系统应支持用户输入自然语言目标，并自动生成结构化 YAML 故事，至少包含：

1. 问题陈述
2. 目标描述
3. 验收准则
4. 约束条件
5. 依赖项
6. 输出要求
7. 风险提示

**关键要求：**

- 生成结果必须可编辑
- 用户可以补充或修正结构化字段
- 结构化故事应能直接作为后续卡片生成输入

### 8.3 自动化泳道处理

系统应支持卡片在以下泳道中自动流转：

1. Backlog Refiner
2. Todo Orchestrator
3. Dev Crafter
4. Review Guard
5. Done Reporter
6. Blocked Resolver

**关键要求：**

- 每个泳道对应明确的提示合同和输出要求
- 每个泳道输出必须结构化
- 卡片只有满足当前泳道输出要求后才可进入下一阶段

### 8.4 人工确认与异常回流

系统应支持以下治理能力：

1. 关键任务人工确认
2. AI 无法继续推进时的异常回流
3. 长时间阻塞后的风险升级
4. 敏感操作的显式确认

**关键要求：**

- 回流原因必须明确记录
- 被回流卡片必须保留上下文和历史轨迹
- 人工接管后可继续原链路推进

### 8.5 外部系统集成

系统应支持通过 MCP、ACP、A2A 和 REST/SSE 与外部平台协作。

外部系统能力包括：

1. 资源系统，提供文档、数据、知识和历史记录
2. 执行系统，完成自动化动作、验证、构建、部署等
3. 上下文系统，维护多步骤会话连续性
4. 协同系统，负责通知、审批和跨节点沟通
5. 治理系统，负责权限、审计、风险和质量控制
6. 集成系统，连接第三方平台和企业内部系统

### 8.6 可视化追踪

系统应提供可视化追踪视图，用于展示：

1. 卡片从创建到完成的流转路径
2. 会话运行轨迹
3. 命令执行记录
4. 文件变更记录
5. 日志和异常信息
6. 关键决策和评审结果

**目标是回答一个问题：到底发生了什么。**

---

## 9. 业务流程

### 9.1 目标发起

发起人创建目标空间，输入目标、范围、约束和验收标准。

### 9.2 卡片生成与分发

KEPLAR 根据目标与上下文生成任务卡片，并分发到对应节点或下游节点。

### 9.3 节点认领与上下文补充

链路用户认领卡片，确认输入是否完整，补充本节点上下文。

### 9.4 AI 泳道加工

Backlog 到 Review 的多个泳道依次完成任务加工、执行和验证。

### 9.5 结果确认与流转

如果满足验收标准，卡片进入 Done 并流向下游节点或归档；如果不满足，进入回流或人工确认。

### 9.6 异常治理

遇到信息不足、执行失败、评审未通过、高风险任务时，系统触发阻塞、升级或回流机制。

---

## 10. 异常分支

### 10.1 信息不足

卡片缺少必要上下文时，系统应回退到发起人或链路用户补充信息。

### 10.2 执行失败

外部系统调用失败、自动化动作失败或工具不可用时，卡片应进入阻塞态并记录原因。

### 10.3 评审未通过

Review Guard 发现验收标准不满足时，卡片应退回前序泳道或人工处理。

### 10.4 高风险操作

涉及敏感资源、关键变更或高影响操作时，系统应强制人工确认。

### 10.5 长时间阻塞

任务长时间停留在同一状态时，系统应触发风险升级，并通知对应责任人处理。

---

## 11. 数据与输出

系统需要生成和保留的核心数据包括：

1. 目标空间信息
2. 节点看板信息
3. 卡片及其状态流转历史
4. YAML 故事
5. 执行简报
6. 实现证据
7. 评审结果
8. 阻塞原因
9. 审计记录
10. 会话轨迹

---

## 12. 成功指标

### 12.1 流程可见性

任务在看板上流转时，团队能够直观看到最新状态、上下文和输出，减少沟通成本。

### 12.2 自动化效率

从目标到可执行故事、从故事到执行准备的时间缩短，卡片生成和推进步骤减少。

### 12.3 质量度量

评审通过率、缺陷率、回流率、阻塞率等指标可衡量流程有效性。

### 12.4 用户采纳率

目标空间创建数、节点看板使用率、活跃用户数、周卡片流转量等指标反映产品价值。

### 12.5 系统可靠性

接口成功率、会话运行成功率、回流恢复率、异常处理成功率等指标反映平台稳定性。

---

## 13. 设计原则

1. **目标优先于任务**
2. **卡片是协作和治理的最小单元**
3. **默认 AI 自主推进，人工只在关键节点介入**
4. **统一目标空间，不统一操作界面**
5. **弱状态管理，强治理兜底**
6. **输出必须结构化，可审计，可复盘**

---

## 14. 待确认事项

1. 首发场景优先面向研发协作还是跨部门通用流程
2. YAML 故事的标准字段是否固定
3. 是否在首版中支持多节点并行
4. 是否需要单独的运维监控面板
5. 外部系统集成首批优先支持哪些协议和平台

## 15. 当前版本产品基线（2026-07-11 审查更新）

本节不替代前文的产品愿景和目标态需求；它定义当前可对外验证的
**Web-first 协作 Beta**，以避免将规划中的 AgentOS 能力误表述为已交付。
完整证据和差异说明见
[`../review/2026-07-11-product-implementation-consistency-audit.md`](../review/2026-07-11-product-implementation-consistency-audit.md)。

### 15.1 当前可验证范围

1. 已认证用户可以创建、管理和查看 Goal Space、Node Board 与 Card。
2. `initiator`、`chain_user` 与 `viewer` 通过资源归属/成员关系获得不同访问和操作权限。
3. Card 在固定状态机内流转；高风险、低置信、外部写相关或不可逆路径通过人工确认门禁治理。
4. 六个 AI 角色以确定性 fixture executor 演示单卡处理；执行、状态转换、确认、审计和 SSE 事件被持久化并可回放。
5. Web 工作台提供目标/任务索引、看板、任务时间线、确认队列和基础创建入口。

### 15.2 当前不应宣称的能力

1. 当前已提供确定性 demo 的 Story 草稿生成、完整 JSON 编辑和显式应用；它不使用真实 LLM，且仅创建初始工作板与 Cards。真实自然语言理解/自动规划仍未实现。
2. 当前角色模型已实现授权和资源可见性；发起人、链路用户、治理人员的专属工作视图尚未冻结为当前验收范围。
3. fixture executor 不会调用 LLM、MCP、ACP、A2A、Git、文件系统或第三方系统；真实外部写仍属后续范围。
4. Tauri/Rust 等价运行时、企业 SSO/多租户、生产高可用和 Kubernetes 不在当前版本范围内。
5. 部分工作台运行时信息（例如 Card runtime、真实 token 计量和右栏 Node Board 信息）仍是延后接线，不应作为产品数据承诺。

### 15.3 当前版本的验收重点

- 人工建立的协作对象在权限、状态机、确认、审计和实时事件约束下可完成可演示闭环。
- 页面展示不得把 placeholder 或 fixture 输出伪装为真实外部执行结果。
- 新增产品入口或真实执行能力前，必须先补充可编辑 Story/草稿 Card、角色视图和人机职责边界的需求与验收条件。

- **编码规范：** Phase 1 前端 TypeScript/React 代码应采用 ESLint + Prettier 统一风格；当前仓库尚未提供 `.eslintrc` 或 Prettier 配置，需在前端代码落地时一并补齐。Rust 代码使用 rustfmt 默认配置和 Clippy 分析。命名方式：TS 侧采用驼峰，数据库字段和 JSON 接口字段采用小写下划线映射（使用 Drizzle 可自动转换）。
- **日志与监控：** Phase 1 必须记录运行时关键事件（目标创建、会话开始/结束、AI 执行结果、人工确认、审计写入、SSE 投递失败），可先使用结构化应用日志。OpenTelemetry 调用链路和指标采集属于后续生产化能力，不作为首轮冻结门禁。
- **配置管理：** Phase 1 至少支持 `DATABASE_URL`、`KEPLAR_DB_DRIVER`、`KEPLAR_DB_PATH`、端口号等环境变量。当前 `docker-compose.yml` 仅提供 PostgreSQL 依赖服务，不代表 Web/API/worker 已具备容器化部署；生产级 Docker/Kubernetes 配置后续冻结。
- **CI/CD：** 当前仓库只有 `.github/workflows/card-verify.yml` 用于卡片 YAML 校验，尚未具备应用测试门禁。Phase 1 应先新增规范一致性、领域单元、API 合同、数据库迁移和实时事件合同测试，再接入 GitHub Actions；E2E、性能测试和发布构建在核心路径稳定后加入。
- **版本管理：** 前端代码遵循 semantic versioning（在 `package.json` 中标注版本，并在发布脚本中同步 npm 包版本）。Rust workspace 版本由根 `Cargo.toml` 统一管理，crate 发布策略在 CLI/API 稳定后再冻结。依赖管理应固定版本号并提交对应 lock 文件以保证可复现性。
- **依赖管理：** 当前 `package.json` 暂无依赖；Phase 1 引入前端、数据库驱动、测试工具或本地扩展时需同步记录用途和安全审计策略。生产部署如果使用 SQLite，需安装对应 SQLite 驱动；Rust 依赖应锁定兼容版本。
- **测试规范：** 当前尚未集成 Vitest、Playwright 或 `api:validate` 类命令。Phase 1 测试基线以 Vitest/TS 领域测试、Drizzle 迁移测试、接口合同测试和 SSE 事件合同测试为先；Playwright E2E 在核心用户流程和服务入口稳定后再加入。

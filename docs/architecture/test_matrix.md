## 3. 核心路径测试 (Happy Path)

### EP-001: 完整流程测试

```
测试名称: EP-001 - 发起人说一句话到卡片完成的全流程
测试步骤:
  1. 发起人输入: "株机厂有一个10列6编组的南京地铁42号线项目开工了，交期是2027年6月"
  2. 系统生成 YAML Story
  3. 发起人确认创建目标空间
  4. 发起人启动会话
  5. AI (Backlog Refiner) 生成任务卡片
  6. AI (Todo Orchestrator) 编排任务
  7. AI (Dev Crafter) 执行任务
  8. AI (Review Guard) 评审任务
  9. AI (Done Reporter) 汇总节点
  10. 发起人查看进度，确认完成

验收点:
  - 每一步都在审计轨迹中有记录
  - 状态变更实时推送到 Dashboard
  - 全部由 AI 自动完成，无需人工介入
```

---

### EP-002: 异常回流测试

```
测试名称: EP-002 - 评审失败后异常回流的完整流程
测试步骤:
  1. 完成 EP-001 的步骤 1-8
  2. Review Guard 评审未通过
  3. 卡片进入 Blocked 状态
  4. Blocked Resolver 分析阻塞原因
  5. 系统通知链路用户
  6. 链路用户修复问题
  7. 卡片重新进入 Review
  8. Review Guard 评审通过
  9. 流程继续

验收点:
  - 评审失败时立即进入 Blocked
  - 阻塞原因清晰可见
  - 修复后流程自动继续
  - 两次评审记录都在审计轨迹中
```

---

### EP-003: 人工介入测试

```
测试名称: EP-003 - 高风险任务的人工确认流程
测试步骤:
  1. Dev Crafter 标记任务为高风险
  2. 系统推送给发起人待确认
  3. 发起人审阅后选择"确认放行"
  4. 任务进入下一阶段

验收点:
  - 高风险任务被正确识别
  - 确认前任务不能流转
  - 人工确认操作被记录
  - 确认后流程自动继续
```

## 5. 测试执行计划

Phase 2 **Web Collaboration Beta** 从已完成的 Web-first baseline 开始，当前根目录可执行验证命令为：

```bash
pnpm typecheck
pnpm lint
pnpm test
pnpm build
pnpm format:check
pnpm check
```

Phase 2 文档和实现应继续以这些命令作为基础门禁，并按功能增量补充 API contract、SSE contract、smoke 和 Playwright happy-path E2E。Tauri、Rust Axum、生产 Kubernetes、企业 SSO 和真实 MCP/ACP/A2A 外部写仍保持延期范围。

### 5.1 测试环境

| 环境 | 用途 | 数据库 |
|------|------|--------|
| Dev | 本地开发 | SQLite |
| Staging | 预发布测试 | PostgreSQL |
| Prod | 生产环境 | PostgreSQL |

### 5.2 测试执行顺序

1. **规范一致性测试** - 校验状态枚举、事件名、字段类型、权限矩阵与文档一致
2. **领域单元测试** - 卡片状态机、目标空间状态机、人工确认门禁
3. **API 合同测试** - 请求/响应结构、错误码、权限拒绝
4. **数据库迁移测试** - Drizzle schema 与迁移可在 SQLite/PostgreSQL 目标下创建
5. **实时事件合同测试** - event envelope、Last-Event-ID replay、重复事件应用和权限过滤
6. **E2E 测试** - 核心路径稳定后再加入
7. **性能测试** - API 和 SSE 实现稳定后再加入

### 5.3 覆盖率要求

| 测试类型 | 覆盖率要求 |
|---------|-----------|
| 单元测试 | >= 80% |
| 集成测试 | >= 60% |
| E2E 测试 | 核心路径覆盖 |

## 6. 第一阶段最小测试门禁

| 门禁 | 覆盖内容 |
|------|----------|
| State enum consistency | Goal Space `draft/active/completed/cancelled` 和 Card `backlog/todo/dev/review/done/blocked/cancelled` 在接口、数据库、状态机中一致 |
| Card state machine | 合法流转、非法流转、Done/Cancelled 终态 |
| Human confirmation gate | pending confirmation 阻断 execute/unblock/complete/外部写操作 |
| Authorization matrix | initiator/chain_user/viewer 端点权限和资源归属 |
| Node board membership | `node_board_members` 控制节点访问；成员增删必须写审计 |
| Audit transaction | 状态变更必须写 transition 和 audit，审计失败则主操作失败 |
| Realtime event contract | 状态变更必须写 realtime event，支持 replay、重复应用和权限过滤 |
| Agent execution contract | `agent_executions.id` 作为 `task_id`，支持状态轮询、重试和权限过滤 |
| Database schema | `goal_spaces/node_boards/node_board_members/sessions/agent_executions/cards/state_transitions/human_confirmations/audit_entries/realtime_events/users` 可创建 |

## 7. CI 备注

现有 `.github/workflows/card-verify.yml` 只校验 `docs/cards/` 下变更的 YAML 文件，并使用 `docs/development-guidelines/card-schema.json`。它是卡片文档校验，不是应用测试基线。
